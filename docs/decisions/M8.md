# M8 — Transactional Outbox

## Goal

M8 closes the at-least-once delivery gap on domain events that ADR-006 explicitly
deferred. ADR-006 records the gap in one line — _"the publish is post-commit but
not transactional"_ — and points to `BACKLOG.md` for the outbox follow-up. M8
promotes that follow-up into a concrete milestone: replace the post-commit
`IPublisher.Publish` loop in `VetCareDbContext.SaveChangesAsync` with an
`OutboxMessage` write inside the same EF transaction, and drain the table with a
hosted background worker, so a process crash between commit and publish no
longer drops the SQS message.

## Scope

- **`OutboxMessage` aggregate.** New table in the `vetcare` schema, written by
  EF Core inside the same transaction as the originating aggregate change.
- **`SaveChangesAsync` becomes the outbox write path.** `VetCareDbContext.SaveChangesAsync`
  collects pending domain events from tracked aggregates (as it does today) and
  writes them as `OutboxMessage` rows in the same transaction, instead of
  dispatching via `IPublisher` post-commit.
- **`OutboxProcessor : BackgroundService`.** New hosted service polls
  unprocessed rows, hydrates the original event type via
  `Type.GetType(message.Type)`, calls `IPublisher.Publish`, marks
  `ProcessedOnUtc` on success, increments `Attempts` and stores `Error` on
  failure with exponential backoff. After a configurable `MaxAttempts` (default
  10) the row is parked as poison and surfaced via a Serilog warning.
- **Stable event metadata on `IDomainEvent`.** `IDomainEvent` gains
  `Guid EventId { get; }` and `DateTime OccurredOnUtc { get; }`, set in the
  event constructor (the current default-implemented getters return fresh values
  on every read, which is unsuitable for outbox round-trips). This absorbs the
  existing backlog item on stable event metadata, since the outbox needs both
  for serialization round-trip and consumer-side idempotency.
- **Tenant context propagation.** `OutboxMessage` carries `TenantId`. The
  worker sets `ITenantProvider` per message before calling `Publish` so
  notification handlers see the correct tenant during replay. Because
  `ITenantProvider` is currently read-only and resolved from `HttpContext`, the
  implementation PR introduces a scoped, mutable provider for the non-HTTP
  worker scope (request-path resolution is unchanged).
- **Integration coverage on the existing Testcontainers Postgres harness.**
  Simulate a publisher failure, assert (a) aggregate persisted, (b) outbox row
  exists with `ProcessedOnUtc == null`, (c) after a retry with a healthy
  publisher, `ProcessedOnUtc` is set and the consumer was invoked exactly once
  for that retry attempt (idempotency expected at the consumer side;
  documented).

## Out of scope

- No changes to `SqsPublisher` itself; the outbox sits between
  `SaveChangesAsync` and the in-process `IPublisher`. Notification handlers
  (`OnAppointmentScheduled`, `OnAppointmentCancelled`) continue to call SQS
  exactly as today.
- No event-payload schema versioning; the assembly-qualified type name is
  enough for the first iteration.
- No outbox-side deduplication; consumers carry the idempotency burden.
- No removal of the inline post-commit `IPublisher.Publish` call until the
  worker is in production and operators have observed at least one full drain
  cycle on real traffic — see "Implementation order" below.

## Data model

- `OutboxMessage`: `Id (Guid PK)`, `OccurredOnUtc (timestamptz)`,
  `Type (text, assembly-qualified)`, `Content (jsonb)`, `TenantId (Guid)`,
  `ProcessedOnUtc (timestamptz, nullable)`, `Error (text, nullable)`,
  `Attempts (int, default 0)`.
- Indexes: `(ProcessedOnUtc, OccurredOnUtc)` for the worker's "next batch"
  scan; `(TenantId)` is not needed at this stage.
- Schema: `vetcare` (consistent with the rest of the model).

## Worker semantics

- **Poll interval.** 2 s default, configurable via an `OutboxOptions` section.
- **Batch size.** 50 default.
- **Retry.** Exponential backoff with a 30 s ceiling per row; total wall-clock
  time to `MaxAttempts` is bounded.
- **Failure handling.** Catch any exception from `IPublisher.Publish`, set
  `Error` to `ex.ToString()` truncated to a sensible length, increment
  `Attempts`, leave `ProcessedOnUtc` null. Once `Attempts >= MaxAttempts`,
  leave the row alone and emit a structured warning. No automatic dead-letter
  move yet.
- **Concurrency.** The worker runs as a single hosted service per app
  instance. `SELECT ... FOR UPDATE SKIP LOCKED` on the batch read so
  multi-instance deploys do not double-publish.
- **Cancellation.** Respect the `CancellationToken` from `ExecuteAsync`; flush
  the in-flight batch before exit.

## Tenant propagation

The worker resolves a per-message `IServiceScope` and sets the
`ITenantProvider` to `OutboxMessage.TenantId` before resolving `IPublisher`.
Notification handlers must not assume an HTTP context — verified against the
two existing handlers:

- `OnAppointmentScheduled` (Application/Appointments/Events) — depends only on
  `ISqsPublisher`, scope-resolvable.
- `OnAppointmentCancelled` (Application/Appointments/Events) — depends only on
  `ISqsPublisher`, scope-resolvable.

Both handlers read `TenantId` from the event payload, not from
`ITenantProvider`, so even without the worker-scoped provider the SQS message
would carry the correct tenant. The scoped provider remains required for
future handlers that may rely on `ITenantProvider` for cross-aggregate reads
during replay.

## Migration plan

- EF migration `AddOutboxMessages` creates the table and indexes in `vetcare`.
- The `IDomainEvent` change is a Domain-layer concern and reaches the four
  existing event types (`AppointmentScheduledEvent`,
  `AppointmentCancelledEvent`, `AppointmentCompletedEvent`,
  `VaccinationRecordedEvent`). The constructor change is a breaking change for
  any external producer of these events; there are none today.
- **Backout.** Drop the migration, remove the worker registration, restore the
  synchronous `IPublisher.Publish` call in `SaveChangesAsync`. No data loss
  because the change is purely additive.

## Test plan

- **Domain unit tests.** `IDomainEvent` constructor sets `EventId` and
  `OccurredOnUtc`; one round-trip serialization test per event type.
- **Application unit tests.** The chosen wiring point (an `OutboxBehavior`, or
  the `SaveChangesAsync` collection step itself, depending on PR (2)) writes
  one `OutboxMessage` per pending domain event during `SaveChangesAsync`;
  happy path and "no events tracked" path.
- **Integration tests** (Testcontainers Postgres + Mongo, real `IMediator`):
  - Scheduled appointment with a failing `ISqsPublisher` substitute → assert
    outbox row + `Attempts == 1` + `Error` populated.
  - Swap to a healthy publisher → assert `ProcessedOnUtc` set on next poll.
  - Swap back to failing publisher and exceed `MaxAttempts` → assert poison
    state + warning logged.
- **End-to-end (Newman).** No new collection step required; the existing
  schedule/confirm/complete flow exercises the worker indirectly via the
  existing reminder queue.

## Acceptance criteria

- All existing 105 tests pass.
- At least 6 new test cases cover the outbox happy path, failure path, retry,
  poison state, tenant propagation, and migration shape.
- `make build` clean, `make lint` clean.
- At-least-once delivery confirmed by the integration test that simulates a
  process crash between commit and publish.
- Branch protection on `vulnerability-audit` left to operator discretion (out
  of scope here).

## Risks

- The `Type`-to-CLR-Type lookup via `Type.GetType(assemblyQualifiedName)`
  breaks if event types are renamed or move assemblies. Mitigation: a small
  registry that maps event type names to types, falling back to
  `Type.GetType` only when not registered. Future-proofing only; not blocking
  M8.
- Long-tail poison rows accumulate in the table. Mitigation: the M8
  monitoring is just a Serilog warning; a future milestone adds a Mongo-audited
  dead-letter table. Out of scope.
- Worker contention with the request hot path on the same DB. Mitigation:
  small batch + indexed scan + 2 s default poll; revisit if pgbench /
  integration timing shows pressure.

## Implementation order (suggested PRs, each landable independently)

1. **`feat/m8-outbox-domain-event-metadata`.** Add `EventId` / `OccurredOnUtc`
   to `IDomainEvent` and the four existing event types; tests; no behavior
   change.
2. **`feat/m8-outbox-table-and-write-path`.** Add `OutboxMessage` + migration
   + the `SaveChangesAsync` change to write rows; **keep** the existing
   post-commit `IPublisher.Publish` call so behavior is unchanged in
   production while the new path bakes.
3. **`feat/m8-outbox-worker`.** Add `OutboxProcessor` and `OutboxOptions`;
   integration tests for happy path and retry.
4. **`feat/m8-outbox-cutover`.** Remove the post-commit `IPublisher.Publish`
   call from `SaveChangesAsync`; the worker becomes the only publish path.
   Stage gate: only after step (3) has been merged and observed for one
   release cycle.
